§1-SEC ACCOUNT-TAKEOVER HARDENING (CRITICAL — DO NOT WEAKEN)

(1) Magic-code email login MUST NOT silently log into an existing password-account by email match.
POST /api/auth/email/verify logic on match:
  - lookup user by email
  - if user exists AND user.email_login_enabled === false:
      → return 403 { error: 'email_login_disabled', hint: 'This account uses password sign-in. Sign in with your password, then enable email sign-in in Account Settings if you want a backup method.' }
  - if user exists AND user.email_login_enabled === true: issue session
  - if user does not exist: create new user with plan='free', email_verified=true, email_login_enabled=true (this email is the primary credential for this brand-new account)

(2) Google sign-in MUST NOT auto-link to an existing password-account by email match.
POST /api/auth/google logic:
  - verify idToken via google-auth-library against env GOOGLE_CLIENT_ID; extract sub, email, email_verified, name, picture
  - if email_verified === false → 403 { error: 'google_email_unverified' }
  - lookup by google_sub → if found, issue session
  - else lookup by email:
      - if found AND user.password_hash IS NOT NULL → 409 { error: 'account_exists_link_required', hint: 'An account with this email already exists. Sign in with your password, then link Google in Account Settings.' }
      - if found AND user.password_hash IS NULL AND user.email_verified → set google_sub on user, issue session (this is a legacy email-only user adding Google)
      - if found AND not email_verified → 409 same as above
  - else: create new user (plan='free', google_sub set, email_verified=true)

Linking Google to an existing password-account ONLY via authed flow: POST /api/account/link/google {idToken} — requires logged-in user, verifies idToken, refuses if google_sub already taken by another user. Same shape for unlink: POST /api/account/unlink/google — refused if it would leave the account without any auth method.

(9) Forced password rotation
- Seeded accounts have password_must_change=true.
- /api/auth/me returns passwordMustChange flag.
- After login, if me.passwordMustChange === true, frontend AuthGate force-redirects to /change-password regardless of intended route, and blocks all other navigation until cleared.
- /change-password page: NO current-password required when passwordMustChange=true (since seed pw is throwaway), min 12 chars, must differ from seed indicator (we don't have the seed pw in plaintext — just check length + not same as username + not in top-100 common-passwords inline list).
- PATCH /api/account/password — if user.password_must_change=true, accept without currentPassword; else require currentPassword. On success set password_must_change=false, insert security_events 'password_changed_forced' or 'password_changed'.

Lockout durations: [60, 300, 900, 3600, 21600, 86400] seconds, saturate at last.
On failure: upsert lockouts(scope='user'|'email', key=identifier) AND lockouts(scope='ip', key=ip); inc fail_count_in_tier; when >=3 → locked_until=now+durations[tier], tier=min(tier+1,5), fail_count_in_tier=0. Successful login resets both scopes. Login endpoints block with 429 {retry_after_seconds} if either active. DB-backed — refresh/cookie-clear cannot bypass.
Extra IP token-bucket on /api/auth/*: 30 req/min/IP.

Routes artifacts/api-server/src/routes/auth.ts:
- POST /api/auth/login {username,password}
- POST /api/auth/google {idToken}
- POST /api/auth/email/request {email} (rate-limit 3/hour/email + IP bucket; reject if email in disposable list; never reveal whether email exists in response)
- POST /api/auth/email/verify {email, code}
- POST /api/auth/logout — revoke current jti
- POST /api/auth/logout-all — revoke all user_sessions of req.user.id, re-issue new session for current device
- GET /api/auth/me → { id, username, email, displayName, avatarUrl, plan, planExpiresAt, isAdmin, passwordMustChange, authMethods: {password, google, email}, currentJti }

Session issuance helper: jti = crypto.randomBytes(16).hex; resolve/create device row by fp_hash; insert user_sessions(user_id, jti, device_id, ip, ua); sign JWT {uid, jti, plan, csrf} 7d exp; set httpOnly Secure(prod) SameSite=Lax cookie `fm_session`; set non-httpOnly `fm_csrf` cookie.

requireAuth middleware: verify cookie, verify jti exists in user_sessions and revoked=false; update last_seen_at; attach req.user. Applied to all /api/* EXCEPT /api/auth/*, /api/healthz, /api/stripe/webhook, /api/config.

CSRF: require header `x-fm-csrf` matching jwt.csrf on mutating endpoints. Skip for /api/stripe/webhook.

Device fingerprint: every successful login → fp_hash = sha256(ip + ua + accept-language + sec-ch-ua); ua_label parsed via ua-parser-js into "{browser} on {os}". Upsert devices. security_events 'login_new_device' if new fp_hash for user.
